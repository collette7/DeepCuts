## Dependency notes

- Plans 001–005 are independently mergeable; none requires another to land first.
- Plan 006 is independent of the search-performance sequence but should preserve the API behavior established by plans 001–005.
- They **compound** for end-to-end latency:
  - 001 (async SDK) — stops one in-flight search from blocking every other endpoint.
  - 002 (background writes) — drops ~100–300 ms of tail latency the user can't see anything during.
  - 003 (pooled httpx) — saves ~0.5–2 s of TLS handshake during the 10-album verification phase.
  - 004 (concurrent auth) — saves ~100–300 ms at the start of every authenticated search.
  - 005 (user-driven retry) — saves the entire **second and third AI calls** when the first attempt returns 7–9 verified albums (the common case). Biggest single win on wall time.
- Recommended order of execution: **001 → 002 → 003 → 004 → 005**. Suggested grouping for review: 001-004 in one PR (purely internal perf, no API contract change), 005 in its own PR (API contract change — adds fields to `SearchResponse`, adds `exclude` to `SearchRequest`).
- 005 has a frontend follow-up that is **explicitly out of scope** here: add a "Find more results" button, POST again with `exclude` populated from already-rendered albums, append the new results. Track it separately; tag any frontend ticket as the consumer of plan 005's response-shape changes.

## How the search path looks today (baseline)

Snapshot of `backend/app/main.py:687–873` at `d9c8a91`:

1. Sync `supabase_admin.auth.get_user(token)` — ~100–300 ms.
2. Sync AI SDK call (`anthropic.Anthropic` / `genai.GenerativeModel.generate_content`) — 1–15 s, blocks event loop. Plan 001 fixes.
3. 10 album verifications in parallel, each opening a fresh `httpx.AsyncClient` — ~50–150 ms TLS overhead × up to 20 calls. Plan 003 fixes.
4. If <10 verified → loop back to step 2, up to 3 times total. Plan 005 removes.
5. Sync Supabase inserts for analytics — ~100–300 ms. Plan 002 fixes.
6. Return response.

After all five plans: one AI call, parallel verifications over a pooled client, analytics deferred, auth resolved opportunistically, no auto-retry.

## Findings considered and rejected

- **Cache repeat queries with Redis (was finding #5).** Rejected for now: the maintainer's call is that users should be able to research / re-query when they want more or higher-quality results — caching could mask that signal and surfaces stale picks. Revisit if analytics ever shows >50% of distinct queries are repeated within an hour.
- **Migrate the server from Hetzner `hel1` (Helsinki) to a US datacenter (was finding #7).** Rejected for now: large effort (full Coolify + Supabase re-deploy + DNS swing + data revalidation), and the geographic latency is only ~3–5% of total search time after the other plans land. Revisit only after 001–005 are deployed and search still feels slow.
